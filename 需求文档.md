# 点餐小程序需求文档

> 版本：v2.0（阶段二已实现：登录+下单+云开发）
> 日期：2026-07-04
> 用途：个人 + 朋友使用，非商业化
> 小程序名称：做饭这件小事

## 一、项目概述

一个「**菜谱 + 点餐下单**」的微信小程序，核心使用场景：

```
朋友来吃饭
   ↓
你把小程序发给朋友
   ↓
朋友浏览菜品、下单（选想吃的菜，不支付）
   ↓
你收到订单，汇总后去采购食材
   ↓
按详情页的做法做菜
```

定位说明：

- **对朋友**：像点餐一样选择想吃的菜。
- **对你**：详情页是菜谱（食材/调味料/做法），订单是你采购的依据。
- 不涉及支付、配送、商业化。

## 二、核心页面结构

小程序共三个页面：

| 页面 | 路径 | 作用 |
| --- | --- | --- |
| 首页（菜品列表页） | `pages/index/index` | 浏览分类与菜品 |
| 商品详情页 | `pages/detail/detail` | 查看某道菜的做法 |
| 我的订单页 | `pages/order/order` | 朋友查看自己的订单；你查看全部订单 |

## 三、首页布局（菜品列表页）

采用「左侧分类 + 右侧列表」结构，从上到下三个区域：

```
┌─────────────────────────────────────┐
│  [头像]  做饭这件小事                │  ← 顶部标题区
│          记录每一道家常菜的做法       │
├──────┬──────────────────────────────┤
│ 荤菜 │  ┌──────────────────────┐    │
│ 素菜 │  │ 菜品卡片：图片+名称   │    │  ← 右侧菜品列表
│ 汤品 │  │            [加入菜单] │    │     每张卡片带「加入菜单」按钮
│ 主食 │  └──────────────────────┘    │
│ 饮品 │  ┌──────────────────────┐    │
│      │  │ 菜品卡片：图片+名称   │    │
│      │  └──────────────────────┘    │
├──────┴──────────────────────────────┤
│  底部：[我的菜单(N)]                 │  ← 浮动按钮，显示已选数量
└─────────────────────────────────────┘
```

### 3.1 顶部标题区

- 左侧：个人图片（占位，后续替换）。
- 右侧：小程序名称「做饭这件小事」+ 简介一句话。

### 3.2 左侧分类栏

- 5 个分类纵向排列：荤菜、素菜、汤品、主食、饮品。
- 当前选中分类高亮。
- 点击切换右侧列表。

### 3.3 右侧菜品列表

- 当前分类下的菜品，卡片纵向排列。
- 每张卡片包含：菜品图片（占位）+ 菜品名称 + **「加入菜单」按钮**。
- 点击卡片本身 → 跳转详情页（看做法）。
- 点「加入菜单」按钮 → 把该菜加入本次点餐清单（不跳转）。

### 3.4 底部浮动按钮

- 显示「我的菜单(N)」，N 为已选菜品数量。
- 点击进入「我的订单页」提交订单。

## 四、商品详情页

展示某道菜的完整做法，按顺序：

1. 菜品图片（顶部，占位）。
2. 菜品名称（标题）。
3. 所需食材（列表，含用量）。
4. 所需调味料（列表，含用量）。
5. 制作过程（有序步骤）。
6. 底部：**「加入菜单」按钮**（与首页一致，可从这里加菜）。

> 详情页只读，不含价格、评价等。

## 五、我的订单页

承担两个作用：朋友提交订单 + 查看订单。

### 5.1 提交订单区（上方）

- 显示当前已加入菜单的菜品列表（名称 + 数量，每道菜 1 份，可移除）。
- 「提交订单」按钮 → 把订单写入云数据库。
- 提交时自动带上：下单人微信昵称、提交时间。

### 5.2 订单列表区（下方）

- 朋友视角：看到**自己**提交过的订单。
- 你（主人）视角：看到**所有人**的订单（用于汇总采购）。

> 「谁是主人」的判定方式见第七节。

## 七、登录与权限

### 7.1 登录方式

- 采用微信「**头像昵称填写能力**」（`open-type="chooseAvatar"` + `<input type="nickname">`）。
  - 原因：微信已停用 `wx.getUserProfile` 返回真实头像昵称，改用填写能力更稳妥。
  - 朋友在订单页主动选头像、填昵称，保存后即可下单。
- openid 由云开发自动获取，作为唯一身份标识，无需用户感知。
- **何时触发**：进入「我的订单页」时若未填昵称，顶部出现填写栏；填完才能提交订单。不强制一进来就登录。

### 7.2 主人判定

- 在云函数 / 小程序配置里写入**主人 openid 白名单**。
- 命中白名单的用户，在「我的订单页」看到所有人的订单；其他人只看自己的。
- 你的 openid 在首次运行后从云开发控制台获取，填入配置即可。

## 八、技术说明

- 框架：原生微信小程序（WXML + WXSS + JS）。
- 后端：微信云开发（云数据库 + 云函数 + 云存储）。
  - 云数据库：存储订单（`orders` 集合）。
  - 云函数：处理「提交订单」「查询订单」（含主人权限判定）。
  - 云存储：后续可上传真实菜品图。
- 菜品数据：本地静态文件，不进云数据库（量小、改动少）。
- 图片：个人头像与菜品图先用占位，留好目录与命名约定。

## 九、已确认决策

1. 小程序名称：**做饭这件小事**
2. 简介：记录每一道家常菜的做法（占位，可改）
3. 分类：荤菜、素菜、汤品、主食、饮品（5 类）
4. 示例菜品：先放几道把流程跑通，后续替换成真实数据
5. 图片：先用占位图，留好目录与命名约定
6. 下单：需要下单功能，**不支付**；每道菜点/不点，不选份数
7. 订单数据存放：**微信云开发**（云数据库），你可在小程序内查看所有人订单
8. 身份识别：**微信授权登录**，订单带昵称
9. 主人判定：openid 白名单，主人看全部订单，朋友看自己的

## 十、本次不做（明确排除）

为避免范围蔓延，以下本次不做：

- 支付、配送。
- 收藏、评价、分享卡片。
- 菜品后台管理界面（直接改本地数据文件）。
- 搜索（菜品不多时不需要）。
- 选份数、备注、口味定制（每道菜点/不点即可）。
- 历史订单统计、采购清单自动汇总（先手动看订单）。

## 十一、开发阶段建议

1. **阶段一**：首页 + 详情页 + 本地菜品数据，跑通浏览与做法查看（不含下单）。
2. **阶段二**：接入云开发，做登录 + 加菜 + 提交订单 + 看订单。
3. **阶段三**：你换真实菜品与图片，获取并填入你的 openid 作为主人。
4. **阶段四**：按使用反馈再迭代（搜索、备注等）。
